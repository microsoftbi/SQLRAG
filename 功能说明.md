## 1. 项目概述

SQLRAG 是一个基于 SQL Server 2025 原生 VectorDB 和 GraphDB 支持的 RAG (Retrieval-Augmented Generation) 系统，提供知识图谱可视化和向量检索问答功能。

**核心特性：**
- 知识图谱可视化与交互
- 基于 LLM 的图数据库问答（含 SQL 自动修正）
- 支持 Legacy 硬编码模式（不依赖 LLM）
- 文档管理与上传（.txt / .md / .docx / .pdf）
- 文档分块（固定分块 / 语义分块）
- 知识库分组管理
- 基于向量检索的智能问答（纯向量检索，无关键词回退）
- Debug 调试工具（执行 SQL / 调用 LLM）

## 2. GraphDB 功能

### 2.1 全景视图

**功能描述：**
展示知识图谱的完整视图，支持节点的点击查看和子图探索。

**主要功能：**
- **图谱可视化**：使用 Vis Network 展示所有节点和边
- **节点分类显示**：不同类型（人物/案件/组织/地点/物品/事件）使用不同颜色
- **点击探索**：点击任意节点弹出子图窗口
- **深度控制**：可调节显示深度（1-5 层），BFS 遍历关联节点
- **动态更新**：调整深度后实时更新子图
- **统计信息**：显示节点数、边数等统计

**操作说明：**
1. 在页面中查看完整的知识图谱
2. 点击任意节点查看详细关联
3. 使用滑块调整显示深度
4. 查看子图的节点和边统计

---

### 2.2 QA 问答

**功能描述：**
基于知识图谱的智能问答系统，支持两种模式：LLM 自动生成 SQL 和 Legacy 硬编码查询。

**主要功能：**
- **问题输入**：支持自然语言问题
- **AI 模式**：调用 LLM 根据 Schema 动态生成 SQL Server 图查询
- **Legacy 模式**：硬编码实体匹配 + 预定义 MATCH 查询模板
- **SQL 展示**：无论 AI 模式还是 Legacy 模式，都在界面展示生成的 SQL
- **自动修正**：AI 模式下 SQL 执行失败时，自动将错误送回 LLM 修正（最多重试 2 次）
- **结果展示**：以易读的文本形式展示查询结果

**使用方式：**
1. 在输入框中输入问题
2. 可选勾选"使用AI生成SQL查询"
3. 点击"提问"按钮
4. 查看生成的 SQL 和查询结果

**示例问题：**
```
方超和周荣是什么关系？
谁认识方超，同时又和周荣有关系？
叶剑遇害案涉及哪些人物？
哪些人物同时出现在多个案件中？
```

**AI 模式技术实现：**
- 读取 `00_GraphDB_SchemaOnly.sql` 作为 Schema 上下文
- 使用 `prompts/graph_sql_generation.md` 作为提示词模板
- `call_llm_generate_sql` → `execute_sql_and_format_result`
- 失败时 `call_llm_fix_sql`（使用 `prompts/graph_sql_fix.md` 修正提示词）
- 执行 SQL 使用 MATCH + CTE + STRING_AGG

**Legacy 模式技术实现：**
- 预定义实体名单（21 人物、5 案件、9 地点、3 组织）
- 关键词模式匹配分派到不同查询函数
- `find_relationship_between_nodes` — 循环查询 5 个 Person→Person 边表
- `_find_related_via_match` — 23 个 UNION ALL 的 CTE 覆盖所有方向边
- `find_common_entities` — 跨案件人物统计

## 3. VectorDB 功能

### 3.1 文档管理

**功能描述：**
文档管理界面，支持查看、添加、删除文档，以及修改文档所属知识库。

**主要功能：**
- **文档列表**：显示所有文档的 ID、标题、所属知识库、来源、创建时间
- **添加文档**：弹窗表单，支持手动添加文档并选择所属知识库
- **删除文档**：点击"删除"，级联软删除文档及其所有分块和向量数据
- **修改知识库**：点击文档所属知识库名称，行内直接切换知识库
- **知识库筛选**：顶部下拉框可按知识库筛选文档列表
- **查看分块**：点击链接查看文档的所有分块

---

### 3.2 文档上传与分块

**功能描述：**
支持多种格式的文档上传，上传后文档暂不入库，用户可选择分块方式并确认后再正式入库。

**主要功能：**
- **拖拽上传**：支持将文件拖拽到上传区域
- **点击上传**：点击区域选择文件
- **格式支持**：.txt, .md, .docx, .pdf
- **上传到知识库**：上传时可指定文档归属的知识库

**上传流程：**
1. 选择或拖拽文件到上传区域
2. 系统自动提取文件内容，仅保存到数据库（不切块、不入库）
3. 在"固定分块"或"语义分块" tab 中选择分块方式
4. 预览分块结果
5. 确认后点击"入库"将分块和向量数据写入数据库

---

### 3.3 固定分块

**功能描述：**
使用 LangChain RecursiveCharacterTextSplitter 按固定字符数进行文本分块。

**主要功能：**
- **Chunk Size**：每个分块最大字符数（默认从系统配置读取，可临时改写）
- **Chunk Overlap**：相邻分块重叠字符数（默认从系统配置读取，可临时改写）
- **分块预览**：点击预览查看分块结果（不入库）
- **入库**：确认后将分块写入 TextChunks 表，并调用 Ollama 生成向量存入 VectorIndex

---

### 3.4 语义分块

**功能描述：**
调用大模型（ChatDeepSeek）对文档进行智能语义分段。

**主要功能：**
- **选择文档**：从已上传的文档列表中选择
- **开始语义分块**：调用 LLM 按语义完整性分段（提示词模板位于 `prompts/vector_semantic_chunk.md`）
- **原文/结果对比**：左侧原文，右侧分块结果（每块带编号和长度）
- **文档入库**：确认后将语义分块结果写入 TextChunks 并生成向量

---

### 3.5 知识库管理

**功能描述：**
对知识库进行增删改查管理，用于对文档进行分组。

**主要功能：**
- **知识库列表**：显示 ID、名称、描述、创建时间
- **创建知识库**：弹窗表单，输入名称和描述
- **编辑知识库**：修改名称和描述
- **删除知识库**：软删除，下属文档的 KnowledgeBaseId 置 NULL

---

### 3.6 系统配置

**功能描述：**
在线查看和修改系统运行参数。

**可配置参数：**
| 参数 | 说明 | 默认值 |
|------|------|--------|
| Chunk Size | 每个文本块的最大字符数 | 500 |
| Chunk Overlap | 相邻文本块重叠字符数 | 100 |
| Embedding Model | Ollama 上的文本嵌入模型 | qwen3-embedding:0.6b |
| Ollama Base URL | Ollama 服务地址 | http://localhost:11434 |

**操作方式：**
1. 修改参数值
2. 点击"保存配置"
3. 即时写入 `.env` 文件并同步到内存，无需重启后端

---

### 3.7 向量索引管理

**功能描述：**
管理 SQL Server 2025 的 VECTOR INDEX。

**重要说明：**
- VECTOR INDEX 存在时，**无法 INSERT 新向量到 VectorIndex 表**
- 正确的操作顺序：**先 DROP INDEX → 嵌入文档 → 再 CREATE INDEX**
- 如果新增文档后检索不到，通常是索引没重建

**操作方式：**
1. 点击"创建索引" — 执行 `CREATE VECTOR INDEX`
2. 点击"删除索引" — 执行 `DROP INDEX`

---

### 3.8 智能问答

**功能描述：**
基于向量检索的 RAG 问答系统，使用纯向量检索召回相关内容，再用 LLM 生成回答。

**主要功能：**
- **问题输入**：自然语言问题
- **向量检索**：使用 `VECTOR_SEARCH(cosine, TOP_N=5)` 进行语义检索
- **来源展示**：显示参考的文档标题和片段
- **答案生成**：使用 LLM 基于检索内容生成回答（提示词模板位于 `prompts/vector_qa_answer.md`）

**问答流程：**
1. 用户输入问题
2. `get_embedding(question)` — 将问题向量化（调用 Ollama）
3. `VECTOR_SEARCH` — 从 VectorIndex 检索 TOP 5 相似向量
4. JOIN TextChunks + Documents — 获取原文片段
5. `generate_answer_with_sources()` — LLM 根据上下文生成回答
6. 返回 answer + sources

**注意：** 不做关键词回退。无检索结果时直接返回空。

## 4. Debug 功能

左侧导航栏点击"Debug"进入调试页面，包含两个 Tab：

### 4.1 SQL 执行

在文本框中输入任意 SQL 语句，点击"执行"，结果以 `el-table` 表格形式展示。

**用途：**
- 测试 GraphDB / VectorDB 的 SQL 查询
- 验证 VECTOR_SEARCH 是否正常工作
- 调试数据问题

### 4.2 LLM 调用

在文本框中输入提示词，点击"调用 LLM"，下方显示模型回复。

**用途：**
- 测试提示词效果
- 验证 LLM 连接和配置
- 调试 prompt 模板

## 5. 后端 API

### 5.1 GraphDB API

#### GET /graph/data
获取知识图谱的所有节点和边数据。

**响应：**
```json
{
  "nodes": [{"id": "...", "type": "Person", "data": {...}}],
  "edges": [{"id": "...", "type": "WorksAt", "from": "...", "to": "...", "data": {...}}]
}
```

#### POST /graph/qa
知识图谱问答接口。

**请求：**
```json
{
  "question": "方超和周荣是什么关系？",
  "use_llm": true
}
```

**响应（AI 模式）：**
```json
{
  "answer": "查询结果...",
  "generated_sql": "WITH person_ids AS ... SELECT ... MATCH ..."
}
```

**响应（Legacy 模式）：**
```json
{
  "answer": "方超 和 周荣 之间的关系：\n方超 ->(RelatedTo)-> 周荣\n方超 ->(SubordinateOf)-> 周荣",
  "generated_sql": "MATCH 图查询 (Person → Person 边表: SubordinateOf, RelatedTo, ...)"
}
```

---

### 5.2 VectorDB API

#### GET /vector/documents
获取文档列表，支持按知识库筛选。

**请求参数：** `?knowledge_base_id=1`（可选）

#### POST /vector/documents
手动添加文档。

**请求：** `{"title": "...", "content": "...", "source": "...", "knowledge_base_id": 1}`

#### PUT /vector/documents/{document_id}
更新文档信息（支持修改所属知识库）。

**请求：** `{"knowledge_base_id": 2}`

#### DELETE /vector/documents/{document_id}
软删除文档及关联分块、向量。

#### POST /vector/documents/upload
上传文档文件。Content-Type: `multipart/form-data`。

**参数：** `file`（文件）, `knowledge_base_id`（可选）

#### GET /vector/documents/{document_id}
获取单个文档详情。

#### GET /vector/documents/{document_id}/chunks
获取指定文档的所有分块。无分块时自动创建。

#### POST /vector/documents/{document_id}/embed
为指定文档的分块生成嵌入向量。

#### POST /vector/documents/{document_id}/preview-chunks
预览固定分块结果（不入库）。

**请求：** `{"chunk_size": 500, "chunk_overlap": 100}`

#### POST /vector/documents/{document_id}/semantic-chunk
调用 LLM 对文档进行语义分块。

**响应：** `{"original_content": "...", "chunks": [...], "total_chunks": N}`

#### POST /vector/documents/{document_id}/commit-chunks
固定分块并入库 + 生成向量嵌入。

**请求：** `{"chunk_size": 500, "chunk_overlap": 100}`

#### POST /vector/documents/{document_id}/commit-chunks-raw
将指定的分块列表直接入库 + 生成向量嵌入（用于语义分块结果）。

**请求：** `{"chunks": ["第一段...", "第二段...", "..."]}`

#### GET /vector/knowledge-bases
获取所有知识库。

#### POST /vector/knowledge-bases
创建知识库。`{"name": "...", "description": "..."}`

#### PUT /vector/knowledge-bases/{kb_id}
更新知识库。

#### DELETE /vector/knowledge-bases/{kb_id}
软删除知识库（关联文档置 NULL）。

#### GET /vector/chunks
获取所有分块列表。

#### POST /vector/qa
向量检索问答（纯 VECTOR_SEARCH，无关键词回退）。

**请求：** `{"question": "用户的问题"}`

**响应：** `{"answer": "生成的回答", "sources": [...]}`

#### GET /vector/config
获取系统配置。

**响应：** `{"chunk_size": 500, "chunk_overlap": 100, "embedding_model": "qwen3-embedding:0.6b", "ollama_base_url": "http://localhost:11434"}`

#### PUT /vector/config
更新系统配置（写入 .env + 同步内存）。

#### POST /vector/index/create
创建 VECTOR INDEX。

#### POST /vector/index/drop
删除 VECTOR INDEX。

## 6. 配置说明

### 6.1 环境变量配置

在 `backend/.env` 文件中配置：

```env
# 数据库配置
SERVER=LOCALHOST
PORT=1433
DATABASE_GRAPH=GraphDB
DATABASE_VECTOR=VectorDB
USER=sa
PASSWORD=Passw0rd

# 后端端口
BACKEND_PORT=8798

# 分块配置
CHUNK_SIZE=500
CHUNK_OVERLAP=100

# Embedding 配置
EMBEDDING_MODEL=qwen3-embedding:0.6b
OLLAMA_BASE_URL=http://localhost:11434

# LLM 配置（兼容 OpenAI / DeepSeek / Volcengine Ark）
LLM_API_KEY=your_api_key_here
LLM_BASE_URL=https://api.deepseek.com/v1
LLM_MODEL=deepseek-chat
```

### 6.2 日志配置

- **日志文件**：`backend/llm_sql_logs.log`
- **日志内容**：API Key（脱敏）、Base URL、模型名、用户问题、生成的 SQL、RAG 回答、文档操作、配置更新等

### 6.3 分块策略

| 方式 | 说明 | 参数来源 |
|------|------|----------|
| 固定分块 | LangChain RecursiveCharacterTextSplitter 按字符数切分 | .env 配置，上传时可临时改写 |
| 语义分块 | 调用 LLM 智能语义分段（`prompts/vector_semantic_chunk.md`） | 需配置 LLM_API_KEY |

## 附录

### A. 嵌入模型说明

| 曾用模型 | 维度 | 说明 |
|----------|------|------|
| quentinz/bge-large-zh-v1.5 | 1024 | 稳定，中文好 |
| bge-m3 | 1024 | 多语言但偶发 NaN 问题 |
| qwen3-embedding:4b | 2560 | 4B 参数，需改 VECTOR 类型 |
| **qwen3-embedding:0.6b** | **1024** | **当前使用，轻量稳定，VECTOR(1024) 直配** |

切换模型后旧向量与新向量不在同一语义空间，需 DROP INDEX → 重新嵌入 → CREATE INDEX。

### B. 数据库表结构

**GraphDB:** Person, CaseNode, Organization, Location, Item, Event（6 节点表）+ 20 边表

**VectorDB:**
- `KnowledgeBases` — 知识库表
- `Documents` — 文档表（FK → KnowledgeBases）
- `TextChunks` — 文本分块表（FK → Documents）
- `VectorIndex` — 向量索引表（FK → TextChunks，`EmbeddingVector VECTOR(1024)`，`idx_content_vector`）

### C. 技术栈详情

**后端：**
- FastAPI 0.109.0 / Uvicorn 0.27.0
- PyODBC 5.1.0（SQL Server）
- LangChain + langchain-deepseek（LLM 集成）
- Ollama 0.1.7（本地嵌入）
- python-docx 1.1.0 / PyPDF2 3.0.1（文档解析）

**前端：**
- Vue 3.4.0 / Vue Router 4.2.5
- Element Plus 2.4.4
- Vite 5.0.0
- Axios 1.6.2
- Vis Network 9.1.6

### D. 注意事项

1. **向量索引写入限制**：VECTOR INDEX 存在时不能 INSERT VectorIndex，需要先 DROP INDEX
2. **模型切换**：切换嵌入模型后需重新嵌入所有文档
3. **NaN 容错**：部分嵌入模型可能产生 NaN，系统会自动截断文本重试，跳过失败分块
4. **SQL 修正**：LLM 生成的 SQL 执行失败时自动重试修正，最多 2 次
