# 2026-06-08 修改汇总

今天主要围绕 **VectorDB 向量检索、GraphDB MATCH 查询、Embedding 模型与索引管理** 做了修改。

## 1. Vector QA 从 LIKE 改成 VECTOR_SEARCH

修改位置：`backend/main.py`

原来 `/vector/qa` 里是用：

```sql
c.ChunkText LIKE ?
```

做关键词匹配。

现在改为：

```sql
VECTOR_SEARCH(
    TABLE = VectorIndex AS v,
    COLUMN = EmbeddingVector,
    SIMILAR_TO = @v,
    METRIC = 'cosine',
    TOP_N = 5
)
```

流程变成：

1. 用户问题通过 `get_embedding(question)` 生成 query vector
2. SQL Server 2025 `VECTOR_SEARCH` 在 `VectorIndex.EmbeddingVector` 上做 cosine 相似度检索
3. 取 TOP 5 个相似 chunk
4. 把 chunk 作为上下文传给 LLM 生成答案

后来因为 `VECTOR_SEARCH` 返回列里没有 `distance` 字段，删除了：

```sql
v.distance
ORDER BY v.distance
```

避免报错：

```text
Invalid column name 'distance'
```

---

## 2. GraphDB 多个方法改成 MATCH 图查询

修改位置：`backend/main.py`

### 2.1 `find_common_entities`

原来用 `$node_id = $from_id`、`$to_id = $node_id` 的普通 JOIN。

现在改成：

```sql
MATCH(p-(edge)->c)
```

通过 CTE + UNION 汇总人物和案件关系。

涉及边：

- `SuspectOf`
- `VictimOf`
- `PerpetratorOf`
- `Witness`
- `Investigates`

---

### 2.2 `find_common_related_entities`

原来循环所有 edge 表，通过：

```sql
STRING_ESCAPE($from_id, 'json') = STRING_ESCAPE(?, 'json')
```

手动比对 graph id。

现在新增了辅助方法：

```python
_find_related_via_match(cursor, person_name)
```

通过 MATCH 查询某个人相关联的所有实体，再在 Python 中取两个实体关联集合的交集。

---

### 2.3 `find_related_entities`

原来也是循环 edge 表 + graph id 字符串比较。

现在复用：

```python
_find_related_via_match(cursor, entity_name)
```

直接用 MATCH 查关联实体。

---

### 2.4 `find_relationship_between_nodes`

原来循环所有边表用 `$from_id/$to_id` 比对。

现在改成对 Person→Person 类型边使用 MATCH：

- `SubordinateOf`
- `RelatedTo`
- `ConflictWith`
- `CooperatesWith`
- `CommunicatesWith`

## 7. 修复 TextChunks 有数据但 VectorIndex 没数据的问题

### 7.1 增加嵌入诊断日志

修改位置：`backend/main.py`

新增日志：

```text
Embedding document X: TextChunks total=?, active=?, VectorIndex total=?, active=?
Embedding document X: found ? chunks without vectors
```

用于判断：

- TextChunks 是否存在
- IsDeleted 是否为 0
- VectorIndex 是否已有记录

---

### 7.2 修复 IsDeleted 为 NULL 导致查不到 chunk

日志发现：

```text
TextChunks total=28, active=0
```

说明插入的 chunk 记录里 `IsDeleted` 不是 0，可能是 NULL。

修复：

```sql
ISNULL(c.IsDeleted, 0) = 0
```

并且插入 TextChunks 时显式写入：

```sql
IsDeleted = 0
```

插入 VectorIndex 时也显式写入：

```sql
IsDeleted = 0
```

---

### 7.3 修复软删除 VectorIndex 记录导致重复入库跳过

原来查询未嵌入 chunk 时：

```sql
LEFT JOIN VectorIndex v ON c.ChunkId = v.ChunkId
```

会把软删除的 VectorIndex 记录也算作已存在。

修复为：

```sql
LEFT JOIN VectorIndex v
ON c.ChunkId = v.ChunkId AND v.IsDeleted = 0
```

并且插入 VectorIndex 时改为：

```sql
IF EXISTS (SELECT 1 FROM VectorIndex WHERE ChunkId = ?)
BEGIN
    UPDATE VectorIndex
    SET EmbeddingVector = @v,
        CreatedAt = GETDATE(),
        IsDeleted = 0
    WHERE ChunkId = ?;
END
ELSE
BEGIN
    INSERT INTO VectorIndex (...)
    VALUES (...);
END
```

这样旧的软删除记录会被恢复，不会违反唯一键。

## 9. 修复长文本 embedding 超上下文问题

遇到错误：

```text
the input length exceeds the context length
```

原来的逻辑是：

```python
max_chars = 1200
truncated_text = text[:max_chars]
```

但如果原文本只有 489 字符，截断 1200 后还是 489，重试无效。

现在改成多级递减重试：

```text
原文
1200 字符
800 字符
500 字符
300 字符
150 字符
```

并跳过长度没有变化的重试。

## 10. 改善 embedding 错误处理

修改位置：`backend/main.py`

### 10.1 单个 chunk 失败不终止整个流程

原来：

```python
raise
```

现在：

```python
continue
```

失败的 chunk 会记录日志，后面的 chunk 继续处理。

---

### 10.2 如果所有 chunk 都失败，返回明确错误

如果：

```python
embedded_count == 0 and failed_errors
```

则抛出明确异常，不再静默返回“成功嵌入 0 个”。

---

### 10.3 commit 接口不再吞掉 embedding 异常

涉及：

- `/vector/documents/{document_id}/commit-chunks`
- `/vector/documents/{document_id}/commit-chunks-raw`

原来 embedding 失败后仍然返回成功，只是 `embedded = 0`。

现在会返回 500：

```json
{
  "message": "分块已创建，但向量嵌入失败: ..."
}
```

方便前端看到真实错误。

## 12. 目前建议的操作流程

现在索引是手动控制，所以建议流程是：

1. VectorDB → 索引管理 → 删除索引
2. 上传文档
3. 文档入库
4. 确认 `VectorIndex` 有数据
5. VectorDB → 索引管理 → 创建索引
6. QA 页面提问测试向量检索



Note:
文档分开太长，会报错。目前来看分在500以内都算安全。
